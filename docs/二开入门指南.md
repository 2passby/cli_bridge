# botmux 二开入门指南

本文不是 README 的重复版，而是基于当前代码结构整理的一份“快速理解项目并开始二开”的开发视角文档。

适合你在下面这些场景使用：

- 想快速知道项目从哪里启动、消息怎么流转、状态存在哪里
- 想加一个新功能，但不知道该改 `daemon`、`worker` 还是 `card-handler`
- 想接入新的 CLI、扩展 Slash Command、改卡片交互、改会话恢复逻辑
- 想避免改着改着把多 bot、tmux、恢复、卡片 PATCH 这些细节搞崩

## 1. 先说结论：这个项目到底在做什么

`botmux` 是一个“飞书话题群 <-> 本地 AI 编程 CLI”的桥接器。

它的核心设计是：

- 每个飞书话题，对应一个独立 session
- 每个 session 背后由一个独立 worker 进程管理
- worker 再去启动一个真实 CLI（Claude Code / Codex / Gemini / CoCo / Aiden / OpenCode）
- daemon 负责：
  - 收飞书消息
  - 建 session
  - 决定是否先选仓库
  - 启停 worker
  - 维护流式卡片
  - 处理 slash command、定时任务、接入现有 tmux 会话
- worker 负责：
  - 启动 CLI
  - 连接 PTY / tmux
  - 维护 Web Terminal
  - 识别“CLI 已空闲可输入”
  - 把终端画面转成文字/截图回传给 daemon

一句话理解：

> `daemon` 是“大脑和编排层”，`worker` 是“每个会话的执行器”，`adapter` 决定“不同 CLI 怎么启动/怎么写入”。

## 2. 先从哪里读代码

如果你第一次看这个项目，建议按下面顺序读：

### 第一层：程序入口

1. `src/cli.ts`
   - 本地命令入口
   - 管理 `setup/start/restart/logs/status/list/delete/schedule/send`
   - 通过 PM2 启多个 daemon 进程

2. `src/index-daemon.ts`
   - daemon 真正启动入口
   - 先加载 `.env`，再动态 import `daemon.ts`

3. `src/daemon.ts`
   - 全项目最核心的编排文件
   - 新消息进来后到底发生什么，基本都在这里

### 第二层：会话执行链路

4. `src/core/session-manager.ts`
   - 会话工作目录、附件下载、prompt 构造、恢复逻辑、定时任务执行

5. `src/core/worker-pool.ts`
   - fork / kill worker
   - streaming card PATCH 串行化
   - CLI skill 安装

6. `src/worker.ts`
   - 单个 session 的执行进程
   - 启 CLI、起 WebTerminal、读写 PTY、做 idle detection、截图上传

### 第三层：飞书侧适配

7. `src/im/lark/event-dispatcher.ts`
   - 飞书 WS 事件分发
   - 群聊是否需要 @mention、bot 是否拥有该 session

8. `src/im/lark/card-handler.ts`
   - 卡片按钮 / 下拉 / TUI prompt 回调

9. `src/im/lark/card-builder.ts`
   - 所有卡片 JSON 在这里生成

10. `src/im/lark/message-parser.ts`
    - 飞书消息解析、附件提取、卡片文本提取

### 第四层：扩展能力

11. `src/core/command-handler.ts`
    - `/close /repo /schedule /adopt /login ...`

12. `src/adapters/cli/*`
    - 各种 CLI 的启动参数、输入方式、技能目录、就绪/完成标记

13. `src/adapters/backend/*`
    - `pty` / `tmux` 两种会话承载方式

14. `src/services/*`
    - session、schedule、message queue、project scanner、frozen card 等本地持久化

## 3. 项目总架构

可以把项目拆成 6 层：

### 3.1 命令层

- `src/cli.ts`
- 负责本地管理员命令
- 不直接处理飞书消息
- 主要做配置管理、PM2 管理、会话列表、botmux 子命令

### 3.2 daemon 编排层

- `src/daemon.ts`
- 负责整个系统状态
- 持有 `activeSessions: Map<string, DaemonSession>`
- 是飞书消息和 worker 之间的总调度器

### 3.3 IM 适配层

- `src/im/lark/*`
- 负责和飞书 API/WS 交互
- 不直接跑 CLI
- 负责事件分发、卡片交互、消息解析、消息发送

### 3.4 session / worker 生命周期层

- `src/core/session-manager.ts`
- `src/core/worker-pool.ts`
- `src/core/command-handler.ts`
- 负责 session 创建、恢复、命令处理、worker fork/kill

### 3.5 执行层

- `src/worker.ts`
- `src/adapters/cli/*`
- `src/adapters/backend/*`
- 负责把“用户消息”真正送进 CLI，并把 CLI 输出变成屏幕内容/截图/WebTerminal

### 3.6 持久化与辅助服务层

- `src/services/*`
- `src/utils/*`
- 负责文件存储、日志、项目扫描、截图渲染、token、队列等

## 4. 核心对象和状态模型

理解二开前，先搞清楚几个核心对象。

### 4.1 `Session`

定义在 `src/types.ts`。

这是“持久化到磁盘”的轻量 session 记录，核心字段包括：

- `sessionId`
- `chatId`
- `rootMessageId`
- `title`
- `status`
- `pid`
- `workingDir`
- `webPort`
- `larkAppId`
- `streamCardId`
- `displayMode`
- `adoptedFrom`

这个对象是“可恢复状态”的来源。

### 4.2 `DaemonSession`

定义在 `src/core/types.ts`。

这是运行时 session，比 `Session` 多很多内存态字段：

- `worker`
- `workerPort`
- `workerToken`
- `pendingRepo`
- `pendingPrompt`
- `pendingFollowUps`
- `streamCardPending`
- `lastScreenContent`
- `frozenCards`
- `tuiPromptCardId`
- `adoptedFrom`

可以理解为：

- `Session` = 持久化快照
- `DaemonSession` = 持久化快照 + 正在运行中的现场

### 4.3 `CliAdapter`

定义在 `src/adapters/cli/types.ts`。

这是适配不同 CLI 的统一接口，核心能力：

- `buildArgs()`
- `writeInput()`
- `passesInitialPromptViaArgs`
- `readyPattern`
- `completionPattern`
- `injectsSessionContext`
- `supportsTypeAhead`
- `skillsDir`
- `altScreen`

二开的关键认知：

- daemon / worker 不需要知道“Claude/Codex/Gemini 具体怎么输入”
- 它们只依赖 `CliAdapter`
- 所以新 CLI 的扩展，优先通过 adapter 完成

### 4.4 `SessionBackend`

定义在 `src/adapters/backend/types.ts`。

抽象“CLI 是运行在什么容器里”的接口：

- `PtyBackend`: 普通 PTY
- `TmuxBackend`: tmux 持久化模式

这层把“CLI 怎么承载”和“CLI 是什么”解耦了。

## 5. 一条消息是怎么走完整链路的

这是最重要的一节。

## 5.1 新话题消息

新消息大概会经历下面的链路：

1. 飞书 WebSocket 收到 `im.message.receive_v1`
2. `src/im/lark/event-dispatcher.ts`
   - 判断是不是群聊
   - 判断是否需要 @bot
   - 判断用户是否在 allowlist
3. 满足条件后，调用 `daemon.ts` 里的 `handleNewTopic()`
4. `handleNewTopic()` 做这些事情：
   - 解析文本和附件
   - 处理 `/xxx` 命令
   - 下载资源到本地
   - 创建 `Session`
   - 创建 `DaemonSession`
   - 扫描仓库
5. 如果找到了多个 repo：
   - 先发 repo 选择卡片
   - `pendingRepo = true`
   - 暂不启动 CLI
6. 如果没有 repo 可选：
   - 直接构造 prompt
   - `forkWorker(ds, prompt)`
7. `worker` 启动后：
   - 起 HTTP + WebSocket terminal
   - 起 PTY/tmux
   - 启 CLI
   - 等 CLI 出现“可输入 prompt”
8. daemon 收到 worker `ready`
   - 发 streaming card
9. worker 周期性把 screen snapshot 通过 IPC 回传
10. daemon PATCH 卡片，用户在飞书看到流式变化

### 5.2 线程内追问消息

线程内回复走 `handleThreadReply()`：

- 先解析文本、附件、合并转发
- 再判断：
  - 是不是 `/close` 这类命令
  - 是不是 OAuth callback
  - 当前 thread 有没有活跃 `DaemonSession`
- 如果 session 还没完成 repo 选择：
  - 消息先缓存在 `pendingFollowUps`
- 如果 session 已经有 worker：
  - 通过 `ds.worker.send({ type: 'message', content })` 送进 worker
- 如果 session 丢了但 thread 还在：
  - 自动补建 session，必要时重新走 repo 选择流程

### 5.3 worker 里的处理

worker 收到 `message` IPC 后：

- 标记新 turn baseline
- 如果是 adopt 模式，直接 raw write
- 否则走 `sendToPty()`
- `sendToPty()` 会根据：
  - 当前 CLI 是否空闲
  - 是否支持 type-ahead
  - 是否被 TUI prompt 阻塞
  决定立刻写还是排队

## 6. repo 选择机制

这是这个项目非常关键的一层 UX。

### 6.1 为什么会有 repo 选择

用户在飞书里开一个话题时，机器人并不知道这次该在哪个仓库里工作。

所以新话题默认不是马上启动 CLI，而是：

1. 根据 `workingDir / workingDirs / projectScanDir` 推导扫描目录
2. 用 `src/services/project-scanner.ts` 扫描 git repo 和 worktree
3. 发 repo 选择卡片
4. 用户点选后才真正 spawn worker

### 6.2 相关字段

`DaemonSession` 里和 repo 选择相关的字段：

- `pendingRepo`
- `repoCardMessageId`
- `pendingPrompt`
- `pendingAttachments`
- `pendingMentions`
- `pendingFollowUps`

### 6.3 用户跳过 repo

用户可以：

- 点卡片里的“直接开启会话”
- 或发 `/skip`

这时会：

- 直接用默认工作目录启动 CLI
- 把之前缓冲的 prompt 和 follow-up 全部打包到 initial prompt

### 6.4 中途切换 repo

如果 session 已经运行中，用户再 `/repo` 选仓库：

- 旧 worker 会被 kill
- 旧 session 标记关闭
- 创建新 session
- 用新的工作目录重新启动 worker

这是“切仓库 = 新会话”的设计。

## 7. streaming card 是怎么工作的

这个项目最复杂、也最容易改坏的部分之一就是 streaming card。

### 7.1 两层卡片

项目里有两类主要卡片：

1. `SessionCard`
   - 更静态
   - 用于打开终端、关闭会话、获取操作链接

2. `StreamingCard`
   - 动态 PATCH
   - 展示当前 turn 的终端内容
   - 支持隐藏 / 文字 / 截图 三种显示模式

### 7.2 streaming card 的显示模式

`DisplayMode` 定义在 `src/types.ts`：

- `hidden`
- `text`
- `screenshot`

卡片按钮逻辑在 `src/im/lark/card-handler.ts`：

- `toggle_display`
- `toggle_mode`
- `term_action`

### 7.3 为什么要有 PATCH 串行队列

飞书 interactive card PATCH 如果并发发送，顺序不可靠。

所以 `src/core/worker-pool.ts` 里实现了：

- `scheduleCardPatch(ds, cardJson)`
- `flushCardPatch(ds)`

特性是：

- 同一 session 同时只允许一个 PATCH 在飞
- 新 PATCH 只保留“最后一个”
- 避免旧状态覆盖新状态

如果你要改 streaming card，一定要尊重这条串行化机制。

### 7.4 为什么会有 frozen card

每个 turn 开始前，旧 turn 的卡片会被冻结成历史卡片。

冻结后依然允许用户点击展开/折叠历史输出，所以有：

- `FrozenCard`
- `services/frozen-card-store.ts`

这也是为什么卡片按钮值里要带 `card_nonce`：

- 当前 turn 的卡片和历史卡片需要区分
- 否则你点历史卡片，可能会改到最新卡片

### 7.5 screenshot 模式的工作方式

当 display mode = `screenshot` 时：

- worker 每 10 秒截图一次
- 用 `captureToPng()` 生成 PNG
- 调用飞书上传图片
- 把 `image_key` 通过 IPC 发回 daemon
- daemon PATCH card 的 img element

截图上传相关逻辑在：

- `src/worker.ts`
- `src/utils/screenshot-renderer.ts`
- `src/utils/lark-upload.ts`

## 8. Web Terminal 是怎么工作的

每个 worker 都会起一个 HTTP 服务，提供 xterm.js 页面。

### 8.1 daemon 视角

daemon 不自己提供 terminal 服务。

它只做：

- 等 worker `ready`
- 拿到 `port + token`
- 拼成：
  - 只读链接：`http://host:port`
  - 可写链接：`http://host:port?token=...`
- 把链接塞到卡片里

### 8.2 worker 视角

worker 里：

- `startWebServer()` 起 HTTP + WebSocket
- 页面内嵌 xterm.js
- 非 tmux 模式：
  - 所有 client 共享 scrollback
- tmux 模式：
  - 每个 WS client 自己 attach 一个 tmux PTY

### 8.3 为什么有 token

terminal 页面默认可能是只读。

只有拿到带 `token` 的链接才允许写入。

这是通过：

- URL query `token`
- worker 内部 `writeToken`
- WS 连接时校验

实现的。

### 8.4 为什么群卡只给“获取操作链接”

群里默认不直接发可写链接，而是提供“获取操作链接”按钮。

按钮逻辑：

- 用户点击卡片
- `card-handler.ts` 识别 `get_write_link`
- 通过飞书 DM 把带 token 的卡片发给点击者

这样更安全，也避免在群里公开写权限。

## 9. idle detection、type-ahead 和 TUI prompt

这是这个项目区别于“简单管道脚本”的关键能力。

### 9.1 idle detection

worker 不能在 CLI 忙的时候随便输入。

所以它需要知道：

- CLI 什么时候已经 ready
- 什么时候一轮输出完成、可继续输入

这由 `src/utils/idle-detector.ts` + `CliAdapter` 提供的模式共同完成。

adapter 可以声明：

- `readyPattern`
- `completionPattern`
- `supportsTypeAhead`

例如：

- Claude Code 支持 type-ahead
- 一些 CLI 不支持，就必须严格等 idle

### 9.2 TUI prompt 检测

某些 CLI 会在终端里弹“交互式选择 UI”，这时直接写文本容易出错。

项目里用了 `ScreenAnalyzer` 做额外识别：

- 周期性分析 screen snapshot
- 检测到 TUI prompt 后
  - 阻塞普通消息输入
  - 发一个飞书卡片，让用户点按钮
- 用户点卡片后，daemon 把选择映射成 `tui_keys` 或 `tui_text_input`
- worker 再把按键序列送入终端

这套逻辑分布在：

- `src/worker.ts`
- `src/utils/screen-analyzer.ts`
- `src/im/lark/card-builder.ts`
- `src/im/lark/card-handler.ts`

如果你后面想增强“终端内菜单交互”，优先看这几处。

## 10. 多 bot 架构怎么实现的

这个项目不是“一个进程管所有 bot”，而是：

> 一个 bot 对应一个 daemon 进程。

### 10.1 配置来源

配置定义在：

- `~/.botmux/bots.json`
- 或 `BOTS_CONFIG` 指定的 JSON 文件

加载逻辑在 `src/bot-registry.ts`。

### 10.2 启动方式

`src/cli.ts` 会读取 bots 配置，生成一个临时 PM2 ecosystem 配置：

- `botmux-0`
- `botmux-1`
- `botmux-2`

每个进程带自己的：

- `BOTMUX_BOT_INDEX`
- `SESSION_DATA_DIR`

然后每个 daemon 启动时只注册自己负责的 bot。

### 10.3 为什么 session key 不是单 rootId

因为多个 bot 可能同时参与同一个飞书线程。

所以运行时 session key 是：

- `sessionKey(rootId, larkAppId)`

而不是单纯 `rootId`。

这能避免：

- 同一 thread 下多个 bot 的 session 冲突

### 10.4 bot-to-bot 协作

项目支持机器人互相 @mention 协作，但飞书不会把 bot 自己发出的消息再推回给所有 bot。

所以项目做了一个本地信号文件机制：

- `botmux send --mention ...`
- 写入 `dataDir/bot-mentions/*.json`
- 目标 daemon watch 这个目录
- 收到后把消息路由给自己的 active session

相关逻辑在：

- `src/cli.ts`
- `src/daemon.ts`

## 11. tmux 模式和 pty 模式

### 11.1 两种后端的区别

#### pty 模式

- 进程退出就没了
- 简单
- 恢复能力较弱

#### tmux 模式

- worker 挂了，tmux session 还在
- daemon 重启后可以重新 attach
- 更适合长期使用

### 11.2 为什么默认会优先 tmux

`src/config.ts` 里 `detectDefaultBackend()` 会检查本机有没有 tmux：

- 有 tmux -> 默认 `tmux`
- 否则 -> `pty`

### 11.3 tmux 模式的重要行为

`TmuxBackend` 的关键设计：

- `kill()` 只是 detach，不销毁 tmux session
- `destroySession()` 才真的 kill tmux session

这意味着：

- daemon 崩了，不代表 CLI 真死了
- `/close` 才是彻底结束

### 11.4 adopt 模式

`/adopt` 可以把“本机已经运行中的 tmux pane”接入到飞书。

流程：

1. `command-handler.ts` 扫描可接入 session
2. 发 adopt 选择卡片
3. 选中后 `startAdoptSession()`
4. 记录 `adoptedFrom`
5. `forkAdoptWorker(ds)`
6. worker attach 到用户原本的 tmux pane

这时 botmux 只是“观察并可转发输入”，不是自己 spawn CLI。

### 11.5 takeover 模式

adopt 后，用户还可以点击“接管”：

- 先 kill adopt worker
- 再把原生 CLI session ID 转成 botmux 标准 session
- fork 标准 worker
- 最后 kill 原 CLI，切换成 botmux 自己控制

这是非常高级的一段流程，改的时候要格外谨慎。

## 12. slash command 能做什么

Slash command 主要集中在 `src/core/command-handler.ts`。

当前支持：

- `/close`
- `/clear`
- `/restart`
- `/status`
- `/help`
- `/cd`
- `/repo`
- `/skip`
- `/cost`
- `/schedule`
- `/login`
- `/adopt`

### 12.1 命令分层思路

这里有两个层次：

1. daemon command
   - 直接由 daemon 截获
   - 例如 `/close` `/repo` `/schedule`

2. botmux CLI subcommand
   - 给 CLI agent 在终端里调用
   - 例如：
     - `botmux send`
     - `botmux schedule add`
     - `botmux thread messages`
     - `botmux bots list`

两者不要混淆：

- Slash command 是“飞书用户输入”的协议
- `botmux xxx` 是“终端内 agent 调用”的协议

## 13. 本地持久化文件都存在哪里

默认都在：

- `~/.botmux/`

更准确地说，运行数据主要在：

- `~/.botmux/data/`

### 13.1 常见文件

- `bots.json`
  - 多 bot 配置

- `.env`
  - 老配置兼容

- `data/sessions.json`
  - 老单 bot session 文件

- `data/sessions-<appId>.json`
  - 当前主流的 per-bot session 持久化文件

- `data/schedules.json`
  - 定时任务

- `data/schedules-output/`
  - 定时任务执行输出

- `data/attachments/<messageId>/`
  - 飞书图片/文件下载落地目录

- `data/bots-info.json`
  - bot open_id、botName、cliId 映射

- `data/bot-openids-<appId>.json`
  - 跨 bot open_id cross-ref

- `data/bot-mentions/*.json`
  - bot-to-bot mention 信号文件

- `data/.botmux-cli-pids/<pid>`
  - CLI pid marker，供子命令推断当前 session

### 13.2 为什么 session-store 要扫描“其他 session 文件”

因为某些子进程里不一定能拿到当前 `larkAppId`。

所以 `session-store.ts` 提供了 fallback：

- 当前 file 里找不到时
- 会扫描所有 `sessions*.json`

这是为了支持从 CLI 子命令里自动推断 session。

## 16. 新增功能前，推荐先确认这几个问题

这是二开时最容易踩坑的点。

### 16.1 你的改动属于 daemon 还是 worker

经验判断：

- 跟飞书消息、卡片、session 路由有关 -> daemon 侧
- 跟 CLI 启动、PTY 输入、截图、terminal 交互有关 -> worker 侧

### 16.2 改动是否要持久化

如果 daemon 重启后也要保留：

- 需要落 `Session`
- 或落 `schedule-store`
- 或落新的 service 文件

否则只改 `DaemonSession` 的内存态就够了。

### 16.3 多 bot 场景是否成立

每次改逻辑都问自己：

- 我是不是误用了全局配置？
- 我拿 session 时是不是只按 `rootId` 查了？
- 这个 open_id 是不是 per-app scoped？

### 16.4 tmux 和 pty 模式都要成立吗

如果你改的是：

- 写入逻辑
- scroll 行为
- attach/detach
- web terminal

一定要想清楚两种 backend 是否都兼容。

### 16.5 这次改动会不会影响 streaming card 并发 PATCH

只要你动了 card 更新逻辑，就考虑：

- 是否可能多次并发 PATCH 同一 message
- 是否应该通过 `scheduleCardPatch()` 走串行队列

### 20.1 优先遵守现有边界

这个项目虽然核心逻辑集中在 `daemon.ts`，但已经明显往模块化拆分了。

## 22. 一张总图

最后给你一个文字版总图，方便建立脑图：

```text
飞书消息
  -> event-dispatcher
    -> message-parser
    -> daemon(handleNewTopic / handleThreadReply)
      -> session-store / message-queue
      -> command-handler / card-handler
      -> session-manager(build prompt / download attachments / restore)
      -> worker-pool(fork worker / patch cards)
        -> worker
          -> cli adapter(build args / write input)
          -> backend(pty or tmux)
          -> idle detector
          -> terminal renderer / screenshot / screen analyzer
          -> IPC back to daemon
      -> card-builder
      -> lark client(send / reply / update / delete)
```

如果你能把上面这条链路记住，二开基本就不会迷路。
